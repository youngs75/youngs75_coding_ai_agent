# Coding AI Agent 개발 요구사항 상세

## 1. 문서 목적과 적용 범위

이 과제의 핵심은 단순히 코드를 생성하는 챗봇을 만드는 것이 아니라,
실제 개발 작업을 지속적으로 수행할 수 있는 Coding AI Agent를 설계하고 구현하는 것입니다.
따라서 기존의 `Skills/SubAgent/Middleware + CRUD/Docker` 실습 흐름은 참고 경로일 뿐,
이번 문서의 핵심 요구사항은 아래 3개로 정의합니다.

1. 장기 메모리와 지식 저장 체계
2. 동적으로 생성되고 정리되는 SubAgent 수명주기 관리
3. Agentic Loop가 깨지지 않도록 하는 복원력과 안전성

### 필수

- 구현 결과는 위 3개 축 각각에 대해 명시적인 설계와 동작 근거를 가져야 합니다.
- 특정 프레임워크의 데모를 재현하는 수준이 아니라, 왜 그 구조가 필요한지 설명 가능한 상태여야 합니다.
- 사용자가 반복 사용하면서 입력하는 지식이 누적되고 다시 활용되는 경로를 포함해야 합니다.

### 권장

- DeepAgents를 가장 먼저 검토하고, 이를 기준선 레퍼런스로 삼아 구조를 설계합니다.
- 구현 코드 주석 또는 README에서 각 요구사항이 코드의 어디에 대응되는지 명시합니다.

### 금지

- 단순 CRUD 코드 생성 데모만으로 과제 목적을 충족했다고 해석하는 것
- 기존 starter template의 TODO를 채우는 것만으로 전체 요구사항을 충족했다고 보는 것
- 특정 제품의 내장 기능을 그대로 사용하는데도, 그 기능의 상태관리나 실패 처리를 설명하지 못하는 것

## 2. 허용 구현 범위

이번 과제는 DeepAgents를 중심 레퍼런스로 권장하지만, DeepAgents만 허용하는 과제는 아닙니다.
Codex, OpenClaw, 자체 LangGraph 구성, 또는 다른 에이전트 프레임워크도 동일 역량을 충족하면 허용됩니다.

### 필수

- 어떤 프레임워크를 사용하든 다음 역량은 동일하게 충족해야 합니다.
  - 장기 메모리의 저장, 조회, 갱신, 정정
  - SubAgent의 런타임 생성과 종료까지 포함한 상태관리
  - 루프 중단, 재시도, fallback, safe stop 중 최소 한 가지 이상의 방어 전략
- 구현은 특정 벤더 UI나 호스팅 서비스에 종속된 블랙박스 기능 설명이 아니라, 저장소 안에서 확인 가능한 구조로 드러나야 합니다.

### 권장

- DeepAgents를 사용할 경우 다음 구성 요소를 비교 기준으로 삼습니다.
  - `MemoryMiddleware` 또는 동등한 메모리 주입 구조
  - `SubAgentMiddleware` 또는 동등한 작업 위임 구조
  - 미들웨어 또는 그래프 노드 수준의 방어 로직
- 대안 구현을 선택했다면 "DeepAgents로 하면 어느 기능에 해당하는지"를 짧게 매핑해 설명합니다.

### 금지

- 특정 제품의 자동 기능을 사용했다는 설명만 있고, 실제로 어떤 이벤트에서 어떤 상태가 기록되는지 말할 수 없는 구현

### 모델 정책

오픈소스 모델 활용은 이번 과제에서 주 권장 경로이며, 제출 프로젝트에서 가산점 요소로 유지합니다.
다만 필수 조건은 아닙니다.

#### 필수

- 어떤 모델을 사용했는지 명시합니다.
- 모델 선택이 요구사항 충족 방식에 어떤 영향을 주는지 설명합니다.
- 모델이나 코드 구성이 tool calling, 긴 문맥, fallback 전환 중 어떤 점에서 제약이 있는지 드러냅니다.

#### 권장

- OpenRouter를 통해 오픈소스 모델을 우선 사용합니다.
- 예시 후보로는 GLM, Nemotron, Qwen 계열처럼 공개적으로 활용 가능한 모델을 우선 검토합니다.
- 프론티어 모델을 쓰더라도, 전체 아키텍처가 해당 모델이 제공하는 전용 기능에 과도하게 잠기지 않도록 설계합니다.

#### 증빙

- `.env`, 설정 파일, README, 또는 실행 로그에서 모델 식별자 확인 가능
- 모델별 fallback 또는 제한 사항 설명

## 3. 핵심 개발 요구사항 3개

### 3-1. 장기 메모리와 지식 저장 체계

장기 메모리는 단순 체크포인트나 대화 히스토리 보존과 다릅니다.
이번 과제에서는 최소한 아래 3개를 구분해야 합니다.

| 메모리 층 | 의미 | 예시 |
| --- | --- | --- |
| `user/profile` | 사용하는 개발자의 선호, 습관, 반복 피드백 | 선호 언어, 코딩 스타일, 자주 쓰는 명령, 싫어하는 출력 형식 |
| `project/context` | 현재 프로젝트의 요구사항, 규칙, 결정 기록 | 아키텍처 결정, 파일 구조, 금지 라이브러리, 테스트 규칙 |
| `domain/knowledge` | 사용자가 앞으로 입력하는 도메인 지식의 누적 저장 | 비즈니스 용어, 업무 규칙, API 계약, 운영 절차 |

#### 필수

- 세 층을 개념적으로라도 분리해 저장해야 합니다.
- 각 층마다 아래를 정의해야 합니다.
  - 무엇을 저장하는가
  - 언제 저장하는가
  - 언제 조회하는가
  - 어디에 지속되는가
  - 잘못된 기억을 어떻게 정정하는가
- `domain/knowledge` 층은 사용자가 도구를 계속 쓰면서 새로 입력하는 지식이 누적되는 구조여야 합니다.
- 메모리는 다음 작업에서 실제로 다시 주입되거나 검색되어야 합니다.

#### 권장

- 메모리 저장소를 파일, SQLite 등으로 분리합니다.
- 메모리 갱신은 명시적 이벤트 기반으로 수행합니다.
  - 사용자 피드백 확정 시
  - 프로젝트 규칙 변경 시
  - 반복해서 등장한 도메인 지식이 확인되었을 때
- 메모리마다 TTL이 아니라 정정 정책을 둡니다. 오래되었다는 이유만으로 무조건 삭제하지 말고, 충돌 시 최신 근거로 교체합니다.

#### 금지

- 체크포인터나 세션 히스토리만 저장하면서, 이를 장기 메모리라고 부르는 것
- 메모리에 저장만 하고, 이후 프롬프트나 추론 과정에서 다시 사용하지 않는 것
- 사용자의 민감 정보, 비밀번호, 토큰, 개인식별정보를 무분별하게 영구 저장하는 것
- 개인화 메모리와 프로젝트 메모리를 섞어 저장해 충돌 원인을 추적할 수 없게 만드는 것

#### 증빙

- 저장소 구조 예시
  - `memory/user_profile.*`
  - `memory/project_context.*`
  - `memory/domain_knowledge.*`
- 메모리 write/read 로그 또는 테스트
- "사용자가 새로운 도메인 규칙을 입력한 뒤, 다음 작업에서 그 규칙을 재사용하는" 시나리오 데모

#### 충족 예시

- 사용자가 "우리 팀은 모든 Python 함수에 타입 힌트를 강제한다"라고 입력하면 `project/context`에 저장되고, 이후 코드 생성에서 자동 반영된다.
- 사용자가 "출력은 항상 한국어 설명 + 영어 코드"를 선호한다고 밝히면 `user/profile`에 저장되고, 다음 세션에서도 유지된다.
- 사용자가 "고객 등급 Silver는 환불 수수료 0%" 규칙을 입력하면 `domain/knowledge`에 저장되고, 이후 결제 로직 생성 시 참조된다.

#### 불충족 예시

- 이전 대화 내용을 그대로 남겨두는 `thread_id` 기반 상태만 있고 별도 메모리 계층이 없다.
- 메모리 파일은 존재하지만 어느 시점에 읽고 쓰는지 코드나 문서에서 확인할 수 없다.
- 사용자 규칙을 저장했지만 이후 생성 결과에 전혀 반영되지 않는다.

### 3-2. 동적 SubAgent 수명주기 관리

동적 SubAgent는 미리 정해 둔 `architect`, `coder` 두 역할을 호출하는 것만으로 충족되지 않습니다.
핵심은 작업의 성격에 따라 SubAgent를 런타임에 만들고, 상태를 추적하고, 정리하는 것입니다.

#### 최소 상태 전이

| 상태 | 의미 |
| --- | --- |
| `created` | SubAgent 인스턴스가 생성됨 |
| `assigned` | 특정 작업, 목표, 제약이 할당됨 |
| `running` | 실제 추론 또는 도구 사용 수행 중 |
| `blocked` | 입력 부족, 도구 실패, 의존 작업 대기 등으로 진행 불가 |
| `completed` | 작업을 정상 완료하고 결과 반환 |
| `failed` | 재시도 후에도 실패 |
| `cancelled` | 상위 오케스트레이터가 중단 |
| `destroyed` | 상태 정리 후 수명주기 종료 |

#### 필수

- SubAgent 생성 조건을 정의해야 합니다.
  - 작업 유형 기준
  - 도구 접근 권한 기준
  - 병렬 처리 필요 여부 기준
- 각 SubAgent는 최소한 아래 메타데이터를 가져야 합니다.
  - `agent_id`
  - `role` 또는 `specialty`
  - `task_summary`
  - `parent_id` 또는 호출자 식별자
  - `state`
  - `created_at`, `updated_at`
- 메인 에이전트와 SubAgent 사이의 결과 반환 규약을 정의해야 합니다.
  - 성공 시 반환 형식
  - 실패 시 반환 형식
  - blocked 시 필요한 추가 입력 형식
- SubAgent 종료 후 정리 정책이 있어야 합니다.
  - 메모리 잔존 여부
  - 로그 보존 범위
  - 임시 리소스 정리

#### 권장

- SubAgent 풀을 고정하지 말고, 역할 템플릿과 현재 작업을 조합해 인스턴스를 만듭니다.
- 실패한 SubAgent를 동일 역할로 재시도할지, 다른 역할이나 다른 모델로 대체할지 기준을 둡니다.
- 부모 에이전트가 SubAgent의 진행 상태를 관찰할 수 있도록 이벤트 로그 또는 상태 저장소를 둡니다.
- 컨텍스트 격리를 기본값으로 두고, 필요한 정보만 명시적으로 전달합니다.

#### 금지

- 미리 선언된 2개 역할만 호출하면서 이를 "동적 생성"이라고 주장하는 것
- SubAgent의 생성과 종료는 있지만, 중간 상태가 전혀 기록되지 않는 것
- 실패했는지, 막혔는지, 완료했는지 구분하지 못하는 것
- SubAgent가 부모의 전체 메시지 히스토리를 무제한 상속받아 컨텍스트 오염이 발생하는 것

#### 증빙

- SubAgent 상태 저장 구조 또는 로그
- `created -> assigned -> running -> ...` 상태가 실제로 찍힌 실행 기록
- blocked, failed, cancelled 중 하나 이상이 드러나는 예외 시나리오 데모

#### 충족 예시

- 사용자가 "문서 요구사항 정리 후 코드 스캐폴딩도 해줘"라고 요청하면, 메인 에이전트가 요구사항 분석용 SubAgent와 구현용 SubAgent를 순차 또는 병렬로 생성하고 각 상태를 기록한다.
- 테스트 실패가 발생하면 기존 coder SubAgent를 무한 재사용하지 않고, reviewer 또는 fixer 역할의 새 SubAgent를 생성해 실패 분석을 맡긴다.
- 도구 권한이 필요한 작업에만 파일 조작 SubAgent를 생성하고, 완료 후 `destroyed`로 전이시켜 임시 작업 디렉터리를 정리한다.

#### 불충족 예시

- `architect`와 `coder`를 코드 상수로 선언하고 언제나 둘 다 한 번씩만 호출한다.
- SubAgent ID나 상태 로그 없이 함수 호출만 한 뒤 결과 문자열만 합친다.
- 실패 시 동일 입력을 같은 SubAgent에 무제한으로 재전달한다.

### 3-3. Agentic Loop 복원력과 안전성

Agentic Loop는 "생각 -> 도구 사용 -> 결과 반영 -> 다음 행동 결정"의 반복 구조입니다. 이번 과제에서는 루프가 멈추거나 깨질 때의 방어 전략을 명시해야 합니다.

#### 필수

- 아래 상황 중 최소 4개 이상에 대해 감지와 대응 정책을 정의해야 합니다.
  - 모델 무응답 또는 과도한 지연
  - 반복 무진전 루프
  - 잘못된 tool call 또는 파싱 실패
  - SubAgent 실패
  - 외부 API 오류
  - 모델 fallback 필요
  - safe stop 필요
- 각 상황마다 아래 정보를 문서화해야 합니다.
  - 감지 신호
  - 허용 재시도 횟수
  - fallback 여부
  - 사용자에게 보여줄 상태
  - safe stop 조건
- 실패가 발생했을 때 최소한 현재 단계, 실패 원인, 다음 행동을 로그나 상태로 남겨야 합니다.
- 시스템이 더 이상 안전하게 진행할 수 없으면, 무리하게 계속하지 말고 안전하게 멈춰야 합니다.

#### 권장

- watchdog 또는 timeout 기반 감지기를 둡니다.
- 동일 모델 재시도와 대체 모델 fallback을 구분합니다.
- 실패 후 재개를 위해 resume metadata를 저장합니다.
  - 마지막 정상 단계
  - 마지막 tool result
  - 마지막 SubAgent 상태
  - 사용자 확인이 필요한 이유
- 루프 중 무한 재계획을 방지하기 위해 max iteration 또는 progress guard를 둡니다.

#### 금지

- 실패 감지 없이 루프를 계속 도는 것
- 오류를 삼키고 정상 완료처럼 보이게 하는 것
- fallback이 불가능한데도 자동 복구가 된 것처럼 설명하는 것
- 실패 후 어떤 상태에서 멈췄는지 남기지 않는 것

#### 증빙

- timeout, retry, fallback, safe stop 중 최소 2개 이상이 보이는 코드 또는 설정
- 실패 시나리오 로그
- 재개 가능 상태 또는 실패 보고 출력 예시

#### 장애 유형별 처리 행렬

| 장애 유형 | 감지 신호 | 허용 재시도 | fallback | 사용자 노출 상태 | safe stop 조건 |
| --- | --- | --- | --- | --- | --- |
| 모델 무응답/지연 | 응답 시간 초과, 스트리밍 중단 | 1~2회 | 가능하면 다른 모델 또는 저성능 안전 모델 | `현재 모델 응답 지연으로 재시도 중` | 재시도 한도 초과 |
| 반복 무진전 루프 | 최근 N회 행동이 동일하거나 결과 변화 없음 | 0~1회 | 계획 축소 또는 다른 SubAgent 투입 | `진전 없음 감지, 전략 전환 시도` | 전략 전환 후에도 진전 없음 |
| 잘못된 tool call | 파라미터 누락, 스키마 불일치, 실행 불가 명령 | 1회 | 도구 호출 재작성 | `도구 호출 형식 오류 수정 중` | 동일 오류 반복 |
| SubAgent 실패 | `failed` 상태 전이, 예외, blocked 장기화 | 역할별 1회 | 다른 역할 또는 다른 모델 SubAgent | `하위 작업 실패, 대체 경로 시도` | 대체 경로도 실패 |
| 외부 API 오류 | 4xx/5xx, rate limit, 네트워크 실패 | 정책 기반 1~3회 | 캐시, 대체 API, 안전 중단 | `외부 서비스 오류 대응 중` | 재시도 비용이 과도하거나 상태 불명확 |
| safe stop 필요 | 권한 부족, 위험 작업, 근거 부족 | 재시도 없음 | 없음 | `안전하게 중단됨. 사용자 확인 필요` | 즉시 중단 |

#### 충족 예시

- 모델이 30초 이상 응답하지 않으면 watchdog이 재시도 1회를 수행하고, 다시 실패하면 fallback 모델로 전환하거나 safe stop 메시지를 남긴다.
- 같은 파일 편집을 3번 연속 실패하면 progress guard가 무한 반복으로 간주하고 루프를 종료한다.
- SubAgent가 `blocked` 상태로 2분 이상 머무르면 부모 에이전트가 추가 입력 요청 또는 취소 결정을 수행한다.

#### 불충족 예시

- LLM 호출 실패 시 예외를 무시하고 빈 문자열을 다음 단계로 넘긴다.
- 같은 계획을 계속 다시 세우기만 하고 종료 조건이 없다.
- 실패 후에도 사용자에게는 "완료"라고 표시된다.

## 4. 권장 기술 경로

이 문서는 도구 중립적 요구사항을 정의하지만,
해당 과정에서 공부한 내용의 특성을 고려하여 다음 경로를 우선 권장합니다.

### 권장 경로 A: DeepAgents 중심

- 오케스트레이터: DeepAgents 또는 LangGraph 기반 메인 에이전트
- 메모리: `MemoryMiddleware` 또는 동등한 메모리 계층 + 별도 저장소
- SubAgent: `task()` 기반 위임 또는 동등한 런타임 SubAgent 생성 구조
- 루프 방어: middleware, graph node, watchdog, timeout, retry policy
- 모델: OpenRouter 경유 오픈소스 모델 우선

### 권장 경로 B: Codex/OpenClaw

- 오케스트레이터: 메인 루프와 상태 머신을 직접 구현
- 메모리: 파일 또는 DB 기반 다층 메모리 저장소
- SubAgent: 역할 템플릿과 인스턴스 상태관리 테이블
- 루프 방어: iteration cap, timeout, failure policy, safe stop
- 모델: 벤더 종속 기능에 잠기지 않는 조합

### 필수

- 어떤 경로를 택하든 요구사항을 우선 충족해야 합니다.
- 권장 경로는 예시일 뿐, 경로를 따랐다는 사실만으로 요구사항 충족으로 간주되지 않습니다.

### 금지

- 특정 프레임워크 이름만 적고 실제 상태관리 설계를 생략하는 것

## 5. 증빙과 검증 기준

이 문서는 채점표 자체는 아니지만, 구현자는 아래 질문에 추가 설명 없이 답할 수 있어야 합니다.

### 필수

- 장기 메모리는 무엇을 어디에 저장하는가
- 메모리는 언제 갱신되고 언제 조회되는가
- 동적 SubAgent는 어떤 조건에서 생성되는가
- SubAgent 상태는 어디에 기록되는가
- SubAgent가 실패하거나 막히면 어떻게 처리되는가
- LLM이 멈췄을 때 무엇을 재시도하고, 언제 멈추는가
- 대체 구현이 허용된다면 어떤 역량을 동등하게 충족해야 하는가
- 오픈소스 모델은 필수인지 권장인지

### 권장

- 각 요구사항별로 아래 네 가지를 같은 형식으로 정리합니다.
  - 설계 의도
  - 코드 위치
  - 로그 또는 데모 시나리오
  - 아직 남아 있는 한계

### 증빙 체크리스트

| 항목 | 최소 증빙 |
| --- | --- |
| 장기 메모리 | 저장 구조 + read/write 시나리오 |
| 동적 SubAgent | 생성/상태 전이/종료 로그 |
| 루프 복원력 | timeout/retry/fallback/safe stop 중 최소 2개 |
| 모델 정책 | 사용 모델과 선택 이유 |
| 대안 구현 정당화 | DeepAgents 기준과의 기능 매핑 |

## 6. 금지 패턴과 오해 방지

아래는 자주 발생하는 오해입니다. 이 항목에 해당하면 요구사항 충족으로 보지 않습니다.

### 금지

- 고정된 2-agent 선언만으로 "동적 SubAgent"를 충족했다고 주장하는 것
- 체크포인터 또는 메시지 히스토리를 장기 메모리와 동일시하는 것
- 메모리 저장만 있고 재활용 경로가 없는 것
- 실패 감지 없이 루프를 계속 도는 것
- 오류 상태를 숨기고 정상 완료처럼 포장하는 것
- 특정 폐쇄형 모델의 능력에 전적으로 기대고, 시스템 설계는 비어 있는 것
- 기존 CRUD/Docker 실습 산출물만으로 이번 요구사항을 모두 충족했다고 보는 것

### 오해 방지 메모

- `thread_id` 유지와 장기 메모리는 다릅니다.
- `architect`, `coder` 역할 분리와 동적 SubAgent 수명주기는 다릅니다.
- 재시도 로직과 fallback 로직은 다릅니다.
- safe stop은 실패가 아니라, 더 위험한 실패를 막기 위한 정상 동작일 수 있습니다.

## 8. 최종 자기 점검

아래 질문에 모두 `예`라고 답할 수 있어야 합니다.

1. 내 시스템에는 `user/profile`, `project/context`, `domain/knowledge`를 구분하는 장기 메모리 설계가 있는가?
2. 사용자가 새 도메인 지식을 입력하면 이후 작업에서 그 지식을 재사용하는가?
3. SubAgent는 런타임에 생성되고, 상태 전이와 종료가 기록되는가?
4. SubAgent가 실패하거나 blocked 되었을 때의 처리 규칙이 있는가?
5. LLM 또는 도구 호출이 실패했을 때 retry, fallback, safe stop 중 무엇을 할지 정의되어 있는가?
6. 시스템이 더 이상 안전하게 진행할 수 없을 때 멈추는 기준이 있는가?
7. 내가 선택한 프레임워크가 DeepAgents가 아니더라도, 동등한 역량을 충족했다는 설명이 가능한가?
8. 오픈소스 모델 사용 여부와 이유를 명시했는가?
9. 기존 3-tier CRUD 실습과 이번 요구사항의 차이를 설명할 수 있는가?

위 질문 중 하나라도 `아니오`라면, 아직 요구사항을 충분히 충족했다고 보기 어렵습니다.
