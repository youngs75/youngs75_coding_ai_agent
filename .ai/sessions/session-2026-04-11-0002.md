## 다음 단계

### 1. 빈 name tool_call 필터링 (최우선)
- `_generate_final`에서 LLM 출력의 tool_calls 중 `name`이 빈 문자열이거나 `arguments`가 빈 객체인 것을 사전 필터링
- 해당 tool_call에 대해 ToolMessage를 생성하지 않고 스킵
- `오류난실행.json`을 테스트 케이스로 활용 가능

### 2. 진척도 체크리스트 주입 시점 개선
- 현재: `loop_idx > 0`이면 무조건 주입 → LLM이 read_file 단계에서도 "빨리 쓰세요" 압박
- 개선안: **첫 write_file 성공 이후**부터 주입, 또는 `written_files`가 1개 이상일 때만 주입
- LLM이 충분히 코드를 파악한 후 자연스럽게 write_file로 전환하도록 유도

### 3. arguments JSON 파싱 방어 강화
- `"{\"path\": \"backend/routers/projects.py\"}}"` 같은 이중 닫기 중괄호 처리
- `tc_args()` 함수에서 JSON 파싱 실패 시 정규식 기반 복구 시도

### 4. Docker 컨테이너 내부 Langfuse 세션 ID 전파
- CLI → Orchestrator → CodingAssistant로 HARNESS_SESSION_ID를 전달하는 메커니즘 필요
- A2A 메시지 metadata 또는 환경변수 주입 방식 검토

### 5. StallDetector read_file 반복 판정 개선
- 현재: 동일 path read_file 2회면 WARN → 의도적 재확인도 stall로 판정
- 개선안: read_file은 3~4회까지 허용, write_file은 동일 path 2회에서 WARN 유지
